# Governance

> [中文版 →](GOVERNANCE.md)

PayDance is currently maintained by one person. The maintainer is responsible for product scope, release timing, licensing, trademarks, and merge decisions.

## Decision Principles

- Protect the core experience first: a local-first, low-distraction desktop real-time salary dashboard.
- Prefer small, verifiable changes with clear maintenance cost.
- Do not add accounts, telemetry, cloud sync, or salary-data uploads by default.
- Keep Windows release quality ahead of platform breadth.
- Time tracking, attendance, payroll, personal finance, and notification-heavy workflows are out of scope by default.

## Contribution Path

1. Small changes may go straight to a PR. Larger features, platform adaptations, or direction changes should start with an Issue that explains the use case and boundary.
2. First contributions are best started from `good first issue` or `help wanted`.
3. Read the [Contributing Guide](CONTRIBUTING_EN.md) before submitting, and include verification that matches the impact.
4. UI, release-chain, security, licensing, and brand changes require explicit maintainer review.

## Maintainer Authority

The maintainer may close issues or pull requests that are out of scope, too risky, insufficiently verified, legally unclear, or too expensive to maintain.

When an idea is worth discussing but not ready for implementation, the maintainer should explain the reason and, where useful, suggest a smaller alternative.

## Governance Changes

Governance can be changed through pull requests. If PayDance gains long-term maintainers, this file should be updated before repository or release permissions are granted.
